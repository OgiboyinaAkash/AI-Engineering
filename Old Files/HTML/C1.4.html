Tool Use & Function Calling

Why LLMs Need Tools

Try this right now: open ChatGPT and ask it "What's the weather in Bangalore right now?"

If you're using an older model or a local LLM, it'll either make something up ("It's about 28°C and sunny!" — sounds great, but it has no idea) or honestly say it doesn't have access to real-time data. Either way, it can't actually check the weather.

Same story with: "What's the price of Bitcoin right now?", "Book me a flight to Mumbai for tomorrow", or "Send Vikram an email saying I'll be late."

Here's the key insight: LLMs can reason about what to do, but they can't do things. They can't browse the internet, they can't query your company's database, they can't call your internal APIs. Their knowledge is frozen at training time and they have no hands to act in the world.

Think of it this way: an LLM is like a brilliant strategist locked in a room with no phone. You slide a note under the door saying "I need to know the weather in Bangalore." The strategist thinks carefully and writes back: "You should call the weather API for Bangalore and check the current temperature." Brilliant advice! But someone else has to actually make the call.

That "someone else" is your code.

This is the idea behind Function Calling (also called Tool Use): instead of the LLM generating a text answer, it outputs a structured request — "I want to call this function with these arguments" — and your code executes that function, gets the result, and feeds it back to the LLM. The LLM then uses the real result to compose a proper answer for the user.

What? That's it? The LLM just... asks us to run a function? Yes! And it's surprisingly powerful.

How Function Calling Works

Let's walk through the entire flow step by step. Say you're building a pizza ordering chatbot and the user says: "Order me a large pepperoni pizza."

Here's exactly what happens:

  1. You define available tools: Before making the API call, you tell the LLM: "Hey, you have access to a function called place_order. It takes a size and a topping as arguments." You describe this using a JSON Schema.
  2. You send the user's query + tool definitions to the LLM.
  3. The LLM decides: "Should I just answer this with text, or do I need to call a tool?" In this case, ordering a pizza clearly needs an action, so it decides to call a tool.
  4. The LLM returns a structured JSON object — NOT a text response. Something like: {"name": "place_order", "arguments": {"size": "large", "topping": "pepperoni"}}
  5. Your code parses this, sees "ah, it wants to call place_order", and actually executes the function. Maybe it hits your restaurant's API, places the order, and gets back: {"order_id": 4521, "status": "confirmed"}.
  6. You send the function result back to the LLM in a follow-up API call.
  7. The LLM composes a natural language response using the real result: "Done! Your large pepperoni pizza has been ordered. Order #4521. It should arrive in about 30 minutes!"

But wait — the LLM didn't actually order the pizza?

Exactly! The LLM just decided what to do and with what arguments. Your code did the actual ordering. The LLM is the brain; your code is the hands.

Before reading on, stop and think about this: why is it designed this way? Why not let the LLM directly execute functions?

The answer is safety. You don't want an LLM autonomously executing database deletes or transferring money without your code validating things first. By keeping execution in your code, you maintain full control.

Here's a summary of the flow:

Step Who Does It What Happens
1 Your Code Sends user query + tool definitions to LLM API
2 LLM Analyzes query, decides to call a tool, returns function name + arguments
3 Your Code Executes the actual function, gets the result
4 Your Code Sends function result back to LLM API (second call)
5 LLM Composes natural language answer using the real result

Notice that we make TWO API calls here. The first one asks the LLM what to do. The second one feeds the tool result back and asks it to compose the final answer. This is important for understanding costs and latency.

Defining Tools for the LLM

Now, how does the LLM know what tools it has access to? You tell it! Every tool is described using a JSON Schema that includes the function's name, what it does, and what parameters it accepts.

Here's what a tool definition looks like for a get_weather function: 

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get the current weather for a given city. Use this when the user asks about weather, temperature, or climate conditions.",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "The city name, e.g. 'Bangalore'"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "Temperature unit. Defaults to celsius."
                    }
                },
                "required": ["city"]
            }
        }
    }
]

And here's the actual Python function that goes with it: 

def get_weather(city, unit="celsius"):
    # In production, this would call a real weather API
    # For now, we return mock data
    weather_data = {
        "Bangalore": {"temp": 28, "condition": "Partly Cloudy"},
        "Mumbai": {"temp": 33, "condition": "Humid"},
        "Delhi": {"temp": 42, "condition": "Sunny"},
    }
    data = weather_data.get(city, {"temp": 25, "condition": "Unknown"})
    return f"{data['temp']}°{'C' if unit == 'celsius' else 'F'}, {data['condition']}"

A few critical things to notice:

  1. The description field is everything. The LLM reads this description to decide when to use the tool. If your description says "Get weather data" but doesn't mention "temperature" or "climate", the LLM might not use it when someone asks "Is it hot in Delhi today?" Be specific and thorough.
  2. Parameter descriptions matter too. The LLM uses them to figure out how to extract the right values from the user's natural language query. If Vikram says "What's the weather like in Bengaluru?", the LLM needs to know that "Bengaluru" maps to the city parameter.
  3. The required array tells the LLM which parameters must be provided. If city is required but the user just says "What's the weather?", the LLM should ask for clarification instead of calling the function without a city.

Think of tool definitions as a restaurant menu for the LLM. If the menu doesn't describe the dishes well, the LLM will order the wrong thing.

The Complete Tool Calling Loop

Let's put it all together with a full working example. This is the code pattern you'll use in almost every tool-calling application: 

import os
import json
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
client = OpenAI()

# --- Step 1: Define the actual function ---
def get_weather(city, unit="celsius"):
    weather_data = {
        "Bangalore": {"temp": 28, "condition": "Partly Cloudy"},
        "Mumbai": {"temp": 33, "condition": "Humid"},
        "Delhi": {"temp": 42, "condition": "Sunny"},
    }
    data = weather_data.get(city, {"temp": 25, "condition": "Unknown"})
    return json.dumps({"city": city, "temperature": data["temp"], "unit": unit, "condition": data["condition"]})

# --- Step 2: Define the tool schema ---
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get the current weather for a given city.",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "The city name"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["city"]
            }
        }
    }
]

# Map function names to actual functions
available_functions = {"get_weather": get_weather}

# --- Step 3: First API call — ask the LLM what to do ---
messages = [
    {"role": "user", "content": "What's the weather in Bangalore right now?"}
]

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=tools,
    temperature=0.0
)

response_message = response.choices[0].message

# --- Step 4: Check if the LLM wants to call a tool ---
if response_message.tool_calls:
    # Append the assistant's response (with tool call info) to message history
    messages.append(response_message)
    
    for tool_call in response_message.tool_calls:
        function_name = tool_call.function.name
        function_args = json.loads(tool_call.function.arguments)
        
        print(f"[Tool Called: {function_name}({tool_call.function.arguments})]")
        
        # Execute the actual function
        result = available_functions[function_name](**function_args)
        
        # Append the tool result to message history
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": result
        })
    
    # --- Step 5: Second API call — get the final answer ---
    final_response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        temperature=0.0
    )
    
    print(f"Assistant: {final_response.choices[0].message.content}")
else:
    # No tool call needed — LLM answered directly
    print(f"Assistant: {response_message.content}")

The output looks like:

[Tool Called: get_weather({"city": "Bangalore"})]
Assistant: The current weather in Bangalore is 28°C with partly cloudy skies.

If you've been following carefully, you'll see the pattern:

  1. We send the query with tools. LLM says "call get_weather with city=Bangalore".
  2. We run get_weather("Bangalore") ourselves and get the result.
  3. We send the result back. LLM writes a nice answer using the real data.

This is the fundamental pattern behind every AI agent, every AI assistant, every tool-using chatbot you've ever seen. ChatGPT browsing the web? Same pattern. Claude running code? Same pattern. It's always: LLM decides → your code executes → LLM responds.

Parallel & Multiple Tool Calls

What happens when Priya asks: "What's the weather in both Bangalore and Mumbai?"

Modern LLMs are smart enough to realize they need to call get_weather twice — once for each city. And they do it in a single response! The tool_calls field in the API response is a list, not a single object. 

# The response might contain:
# response_message.tool_calls = [
#     ToolCall(function=Function(name="get_weather", arguments='{"city": "Bangalore"}')),
#     ToolCall(function=Function(name="get_weather", arguments='{"city": "Mumbai"}'))
# ]

# Our code from the previous section already handles this!
# The for loop iterates through ALL tool calls:
for tool_call in response_message.tool_calls:
    function_name = tool_call.function.name
    function_args = json.loads(tool_call.function.arguments)
    result = available_functions[function_name](**function_args)
    messages.append({
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": result
    })

Notice that each tool call has a unique tool_call.id. When we send the results back, we include this ID so the LLM knows which result belongs to which call.

Here's where it gets interesting: the LLM also knows when it doesn't need tools. If Priya asks "What's the capital of France?", the LLM will just answer directly — no tool call, no second API request. It only invokes tools when it genuinely can't answer from its training data.

Modern models like GPT-4o, Claude 3.5 Sonnet, and Gemini are very good at this decision-making. They'll correctly pick between answering directly, calling one tool, or calling multiple tools in parallel.

Lab: Build a Tool-Using Assistant

1. Problem Setup

You are building a CLI assistant that has access to 3 tools:

  • get_weather(city) — returns mock weather data for Indian cities
  • calculate_tip(bill_amount, tip_percent) — calculates the tip and total
  • search_contacts(name) — looks up a person in a mock contacts dictionary

The assistant should handle queries like "What's the weather in Delhi?", "How much tip on a 450 rupee bill at 15%?", and "Find Priya's phone number" — all through function calling. It should also handle regular questions (like "What's 2+2?") without invoking any tools.

2. Environment Setup

Using the same environment from earlier modules:

pip install openai python-dotenv

3. Step-by-Step Instructions

  1. Define the three Python functions with mock data.
  2. Write the corresponding tool definition schemas (with clear descriptions!).
  3. Implement the tool dispatch loop: detect tool calls → execute the right function → send results back.
  4. Handle the case where no tool is needed (the LLM just answers directly).
  5. Print which tool was called and with what arguments for transparency.
  6. Wrap everything in an interactive loop that reads user input from the terminal.

4. Starter Code

import os
import json
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
client = OpenAI()

# --- Mock Functions ---
def get_weather(city):
    weather_db = {
        "Delhi": {"temp": 42, "condition": "Sunny and hot"},
        "Mumbai": {"temp": 33, "condition": "Humid with light rain"},
        "Bangalore": {"temp": 28, "condition": "Partly cloudy"},
        "Chennai": {"temp": 36, "condition": "Hot and humid"},
        "Kolkata": {"temp": 34, "condition": "Overcast"},
    }
    data = weather_db.get(city, {"temp": 25, "condition": "Data not available"})
    return json.dumps({"city": city, "temperature_c": data["temp"], "condition": data["condition"]})

def calculate_tip(bill_amount, tip_percent):
    tip = bill_amount * (tip_percent / 100)
    total = bill_amount + tip
    return json.dumps({"bill": bill_amount, "tip_percent": tip_percent, "tip_amount": tip, "total": total})

def search_contacts(name):
    contacts = {
        "Priya": {"phone": "+91-98765-43210", "email": "priya@example.com"},
        "Vikram": {"phone": "+91-91234-56789", "email": "vikram@example.com"},
        "Ravi": {"phone": "+91-87654-32100", "email": "ravi@example.com"},
    }
    result = contacts.get(name)
    if result:
        return json.dumps({"name": name, "phone": result["phone"], "email": result["email"]})
    return json.dumps({"error": f"No contact found for '{name}'"})

# --- Tool Definitions ---
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather conditions for an Indian city. Use when the user asks about weather, temperature, or climate.",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name, e.g. 'Bangalore'"}
                },
                "required": ["city"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "calculate_tip",
            "description": "Calculate the tip amount and total bill. Use when the user asks about tips, gratuity, or bill splitting.",
            "parameters": {
                "type": "object",
                "properties": {
                    "bill_amount": {"type": "number", "description": "The bill amount in rupees"},
                    "tip_percent": {"type": "number", "description": "The tip percentage, e.g. 15 for 15%"}
                },
                "required": ["bill_amount", "tip_percent"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "search_contacts",
            "description": "Search for a person's contact information (phone and email). Use when the user asks to find or look up someone's contact details.",
            "parameters": {
                "type": "object",
                "properties": {
                    "name": {"type": "string", "description": "The person's first name"}
                },
                "required": ["name"]
            }
        }
    }
]

# Map names to functions
available_functions = {
    "get_weather": get_weather,
    "calculate_tip": calculate_tip,
    "search_contacts": search_contacts,
}

def chat(user_input):
    messages = [
        {"role": "system", "content": "You are a helpful assistant with access to tools. Use them when needed."},
        {"role": "user", "content": user_input}
    ]

    # First API call
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        tools=tools,
        temperature=0.0
    )

    response_message = response.choices[0].message

    # Check if tools were called
    if response_message.tool_calls:
        messages.append(response_message)

        for tool_call in response_message.tool_calls:
            fn_name = tool_call.function.name
            fn_args = json.loads(tool_call.function.arguments)
            print(f"[Tool Called: {fn_name}({tool_call.function.arguments})]")

            result = available_functions[fn_name](**fn_args)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result
            })

        # Second API call with tool results
        final = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            temperature=0.0
        )
        print(f"Assistant: {final.choices[0].message.content}")
    else:
        print(f"Assistant: {response_message.content}")

if __name__ == "__main__":
    print("Assistant ready. Type 'exit' to quit.\n")
    while True:
        try:
            user_text = input("You: ")
            if user_text.lower() == "exit":
                break
            if not user_text.strip():
                continue
            chat(user_text)
            print()
        except (KeyboardInterrupt, EOFError):
            break

6. Failure Scenario

Try this: remove the description field from the calculate_tip tool definition (or replace it with something vague like "A math function"). Now ask: "How much tip should I leave on a 500 rupee bill at 10%?"

Observe how the LLM might try to answer the question directly using mental math instead of calling the tool, or worse, call the wrong tool entirely. This demonstrates why clear, specific tool descriptions are critical.
