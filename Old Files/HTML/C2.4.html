AI Workflow Orchestration, Tool Integration & MCP

So far, we've built everything from scratch — tool calling loops, RAG pipelines, agent loops. That's essential for understanding what's happening under the hood. But in production, you don't want to re-implement the ReAct loop for the 15th time, or hand-wire ChromaDB integrations into every new project.

This module is about frameworks and protocols that do the plumbing for you, so you can focus on the application logic. Think of it like this: in C2.3, you learned to build an engine from parts. Now you're learning to use a car — same engine underneath, but with a steering wheel and dashboard.

LangChain — Chains and LCEL

LangChain is the most popular framework for building LLM applications. It provides pre-built components for prompts, chains, memory, agents, and retrieval — and a way to compose them together.

The modern way to build with LangChain is LCEL (LangChain Expression Language) — a pipe-based syntax for chaining operations: 

# pip install langchain langchain-openai

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# Define components
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant that translates {input_language} to {output_language}."),
    ("user", "{text}")
])

model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = StrOutputParser()

# Chain them together with the pipe operator
chain = prompt | model | parser

# Run it
result = chain.invoke({
    "input_language": "English",
    "output_language": "Hindi",
    "text": "Good morning, how are you?"
})

print(result)  # "सुप्रभात, आप कैसे हैं?"

See that | pipe operator? That's LCEL. It reads left to right: take a prompt template, pipe it into the model, pipe the output into a string parser. Clean, composable, reusable.

Multi-Step Chains

The real power of chains is composing multiple LLM calls into a workflow: 

from langchain_core.runnables import RunnablePassthrough

# Step 1: Summarize a document
summarize_prompt = ChatPromptTemplate.from_template(
    "Summarize this document in 3 bullet points:\n{document}"
)
summarize_chain = summarize_prompt | model | parser

# Step 2: Generate quiz questions from the summary
quiz_prompt = ChatPromptTemplate.from_template(
    "Based on this summary, create 3 quiz questions:\n{summary}"
)
quiz_chain = quiz_prompt | model | parser

# Compose: document → summary → quiz questions
full_chain = (
    {"summary": summarize_chain, "document": RunnablePassthrough()}
    | quiz_chain
)

result = full_chain.invoke({"document": "CloudNest offers three hosting plans..."})
print(result)

LangChain Memory

Remember from C1.3 how you had to manually manage message history for multi-turn conversations? LangChain handles this with memory objects: 

from langchain.memory import ConversationBufferMemory
from langchain.chains import ConversationChain

memory = ConversationBufferMemory()
conversation = ConversationChain(llm=model, memory=memory)

print(conversation.predict(input="My name is Vikram."))
print(conversation.predict(input="What's my name?"))
# It remembers! No manual message history needed.

LlamaIndex — Purpose-Built for RAG

While LangChain is a general-purpose LLM framework, LlamaIndex (formerly GPT Index) is purpose-built for RAG. It handles document loading, chunking, indexing, and querying with minimal code. 

# pip install llama-index llama-index-llms-openai llama-index-embeddings-openai

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# Load documents from a directory — handles PDF, TXT, DOCX, etc.
documents = SimpleDirectoryReader("./company_docs/").load_data()

# Build index (auto-chunks, auto-embeds, auto-stores)
index = VectorStoreIndex.from_documents(documents)

# Query — one line!
query_engine = index.as_query_engine()
response = query_engine.query("What is the refund policy for enterprise customers?")
print(response)

That's three lines to go from a folder of documents to a working RAG system. Compare that to the 50+ lines we wrote in C1.5 and C2.1. LlamaIndex does the chunking, embedding, vector storage, retrieval, and prompt construction under the hood.

So why did we build it manually first? Because you need to understand what LlamaIndex is doing. When retrieval quality is bad, you need to know whether the problem is chunking, embedding, retrieval top-k, or the prompt. You can't debug a black box.

When to Use Which?

Use Case LangChain LlamaIndex
RAG over documents Possible but verbose Purpose-built. Best choice.
Multi-step chains LCEL makes this elegant Possible but not the primary focus
Agent loops Strong agent framework Has agents but LangChain's are more mature
Conversation memory Built-in memory types Available but simpler
Document ingestion (PDF, DOCX) Needs extra loaders Built-in. Handles 20+ formats.

Model Context Protocol (MCP)

Here's a problem that gets worse as AI systems grow: every LLM provider has its own tool calling format. OpenAI has one JSON schema. Anthropic has another. Google Gemini has yet another. If you build 5 tools for OpenAI, you can't just plug them into Claude without rewriting the schemas.

The Model Context Protocol (MCP) is a standard created by Anthropic that solves this. It defines a universal way for AI applications to connect to data sources and tools, regardless of which LLM is being used.

Think of MCP like USB for AI tools. Before USB, every device had a different connector. MCP gives tools a standard interface so any AI application can use them.

MCP Architecture

flowchart LR subgraph Client [AI App Client] A[Claude / GPT-4 / Gemini] end subgraph Server [MCP Server] B[Weather API / Database / Files] end Client <-->|MCP Protocol JSON-RPC| Server

Key concepts:

  • MCP Server: A lightweight process that wraps your tools. It exposes them via a standard JSON-RPC interface.
  • MCP Client: Your AI application. It discovers available tools from the server and uses them.
  • Transport: MCP servers can communicate over stdio (local) or HTTP/SSE (remote).

Building a Simple MCP Server

# pip install mcp

from mcp.server import Server
from mcp.types import Tool, TextContent
import json

server = Server("weather-server")

@server.tool()
async def get_weather(city: str) -> list[TextContent]:
    """Get current weather for a city."""
    weather_data = {
        "Bangalore": {"temp": 28, "condition": "Partly Cloudy"},
        "Mumbai": {"temp": 33, "condition": "Humid"},
    }
    data = weather_data.get(city, {"temp": 25, "condition": "Unknown"})
    return [TextContent(type="text", text=json.dumps(data))]

@server.tool()
async def calculate(expression: str) -> list[TextContent]:
    """Evaluate a mathematical expression safely."""
    try:
        # Only allow safe math operations
        result = eval(expression, {"__builtins__": {}}, {})
        return [TextContent(type="text", text=str(result))]
    except Exception as e:
        return [TextContent(type="text", text=f"Error: {e}")]

if __name__ == "__main__":
    import asyncio
    from mcp.server.stdio import stdio_server

    async def main():
        async with stdio_server() as (read, write):
            await server.run(read, write)

    asyncio.run(main())

Once your MCP server is running, any MCP-compatible client (Claude Desktop, VS Code with Copilot, custom applications) can discover and use your tools — without you writing provider-specific integration code.

This is a newer protocol and the ecosystem is still growing. But it's worth learning now because it's becoming the standard for tool integration. Claude Desktop, Cursor, and many IDE plugins already support MCP natively.

Multi-Agent Systems

In C2.3, we built single agents. But some tasks are better handled by multiple specialized agents working together. The most practical pattern is coordinator + specialist:

flowchart TD A[User Task] --> B[Coordinator Agent] B -->|Delegates Research| C[Research Specialist] C -->|Returns Data| B B -->|Delegates Writing| D[Writing Specialist] D -->|Returns Draft| B B --> E[Final Answer]

Here's a simplified implementation:

def run_specialist(specialist_name, system_prompt, task):
    """Run a specialist agent with a specific role"""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": task}
        ],
        temperature=0.0
    )
    return response.choices[0].message.content

def coordinator(user_task):
    """Coordinator breaks task into subtasks and delegates"""

    # Step 1: Research
    research = run_specialist(
        "researcher",
        "You are a research analyst. Find key facts and data points.",
        f"Research the following: {user_task}"
    )
    print(f"[Researcher]: {research[:100]}...")

    # Step 2: Writing
    report = run_specialist(
        "writer",
        "You are a technical writer. Create clear, well-structured reports.",
        f"Write a report based on this research:\n{research}"
    )
    print(f"[Writer]: {report[:100]}...")

    return report

result = coordinator("Compare CloudNest pricing to AWS and DigitalOcean")

When multi-agent is justified: When tasks are naturally decomposable into specialized roles (research + writing, analysis + visualization, coding + testing). When it's overkill: When a single agent with good tools can handle the task. Don't add complexity for complexity's sake.

Prompt Versioning & Management

In production, your prompts are code. They need version control, testing, and documented iterations — just like any other software component. 

import json
import datetime

class PromptRegistry:
    def __init__(self, registry_file="prompt_registry.json"):
        self.file = registry_file
        self.prompts = self._load()

    def _load(self):
        try:
            with open(self.file) as f:
                return json.load(f)
        except FileNotFoundError:
            return {}

    def _save(self):
        with open(self.file, "w") as f:
            json.dump(self.prompts, f, indent=2)

    def register(self, name, version, template, notes=""):
        key = f"{name}:{version}"
        self.prompts[key] = {
            "name": name,
            "version": version,
            "template": template,
            "notes": notes,
            "created": datetime.datetime.now().isoformat(),
            "active": True
        }
        self._save()
        print(f"Registered: {key}")

    def get(self, name, version="latest"):
        if version == "latest":
            versions = [k for k in self.prompts if k.startswith(f"{name}:")]
            if not versions:
                raise KeyError(f"No prompt found: {name}")
            key = sorted(versions)[-1]
        else:
            key = f"{name}:{version}"
        return self.prompts[key]["template"]

    def list_versions(self, name):
        versions = {k: v for k, v in self.prompts.items() if k.startswith(f"{name}:")}
        for key, data in versions.items():
            print(f"  {key} | {data['notes']} | {data['created'][:10]}")

# Usage
registry = PromptRegistry()

registry.register("rag_system", "v1.0",
    "You are a helpful assistant. Answer questions based on the context provided.",
    notes="Initial baseline")

registry.register("rag_system", "v1.1",
    "You are a helpful assistant. Answer ONLY based on the context below. If unsure, say 'I don't know.'",
    notes="Added grounding instruction — accuracy improved 60% → 85%")

registry.register("rag_system", "v1.2",
    "You are a CloudNest support assistant. Answer ONLY from the context. Be concise (max 2 sentences). Cite the source document.",
    notes="Added role, conciseness, citations — accuracy 85% → 93%")

print("\nAll versions:")
registry.list_versions("rag_system")

# Use in production
current_prompt = registry.get("rag_system", "latest")
print(f"\nActive prompt: {current_prompt}")

This registry pattern gives you a clean way to track prompt evolution, roll back to previous versions if a new prompt degrades quality, and A/B test different prompt versions in production.

4. Starter Code

import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
import chromadb

load_dotenv()

# --- ChromaDB Setup ---
chroma = chromadb.Client()
collection = chroma.create_collection(name="cloudnest_kb")
docs = [
    "CloudNest Pro plan costs ₹1,499/month with 2 vCPUs and 4 GB RAM.",
    "Enterprise plan includes dedicated account manager and 24/7 priority support.",
    "All plans include free SSL certificates and daily backups.",
    "Refund window: Starter 14 days, Pro 30 days, Enterprise 90 days.",
    "Data centers in Mumbai, Bangalore, and Singapore.",
]
collection.add(documents=docs, ids=[f"d{i}" for i in range(len(docs))])

# --- Retrieval Function ---
def retrieve(question: str) -> str:
    results = collection.query(query_texts=[question], n_results=3)
    return "\n---\n".join(results["documents"][0])

# --- LangChain LCEL Chain ---
model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = StrOutputParser()

rag_prompt = ChatPromptTemplate.from_template(
    """Answer based ONLY on this context. If unsure, say "I don't know."

Context:
{context}

Question: {question}
Answer:"""
)

rag_chain = (
    {"context": lambda x: retrieve(x["question"]), "question": lambda x: x["question"]}
    | rag_prompt
    | model
    | parser
)

# --- Test ---
test_questions = [
    "How much does the Pro plan cost?",
    "Where are the data centers?",
    "What is the refund window for Enterprise?",
    "Does CloudNest offer GPU instances?",
    "What support does Enterprise get?",
]

print("--- LangChain LCEL RAG Chain ---\n")
for q in test_questions:
    answer = rag_chain.invoke({"question": q})
    print(f"Q: {q}")
    print(f"A: {answer}\n")
