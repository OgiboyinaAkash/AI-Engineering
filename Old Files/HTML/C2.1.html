Production RAG Architecture

Welcome to Track 2. If Track 1 was about learning to cook individual dishes — making an omelette (API calls), baking bread (prompts), preparing a salad (RAG) — then Track 2 is about running a restaurant kitchen. Same ingredients, completely different game.

In C1.5, you built a mini-RAG pipeline with 10-12 hardcoded policy strings. It worked beautifully in the lab. But try scaling that to 500 real documents — messy PDFs, overlapping topics, inconsistent formatting — and things fall apart fast. Which chunks does the retriever miss? Why does the LLM hallucinate on question 47 but nail question 48? Why is the answer weirdly cut off?

This module is about building RAG that actually works in production.

RAG vs Fine-Tuning: When to Use What

Before we go deeper into RAG, let's address the elephant in the room: "Why not just fine-tune the model on our company's data?"

Fine-tuning means taking a pre-trained LLM and training it further on your own data so it "knows" your content natively. It sounds great in theory. In practice, here's why RAG wins for most product teams:

Factor RAG Fine-Tuning
Data freshness Update documents anytime — retrieval sees changes instantly. Must retrain the model every time data changes. Expensive.
Cost Embedding costs are tiny. Generation costs are the same. Training runs cost $100–$10,000+ depending on model size.
Hallucination control Grounding instruction + source citations. Controllable. Model may still hallucinate — you can't easily verify what it "learned".
Infrastructure A vector DB + API calls. Any engineer can set it up. GPU clusters, training pipelines, ML ops. Needs a specialized team.
Best for Knowledge Q&A, document search, customer support, internal tools. Changing the model's tone, style, or behaviour. Domain-specific jargon.

Bottom line: For this program, and for the vast majority of product teams, RAG is the answer. Fine-tuning is a specialised tool for specific problems. When your team has no training infrastructure, no ML engineers, and data that changes weekly — RAG is not just easier, it's better.

Chunking Strategies — The Make-or-Break Decision

In C1.5, we mentioned that chunk size matters. Now let's get serious about it. In production RAG, chunking is often the single biggest lever for retrieval quality. Get it wrong, and your retrieval will feed the LLM irrelevant garbage no matter how good your embeddings are.

Strategy 1: Fixed-Size Chunking

The simplest approach — split text every N characters or tokens. Fast to implement, but blind to meaning boundaries. 

def fixed_size_chunk(text, chunk_size=500, overlap=50):
    chunks = []
    start = 0
    while start < len(text):
        end = min(start + chunk_size, len(text))
        chunks.append(text[start:end])
        start = end - overlap
    return chunks

Problem: This might split "The refund window is" into one chunk and "60 days for premium members" into the next. The retriever finds the second chunk but it lacks context about what has a 60-day window.

Strategy 2: Sentence-Aware Chunking

Split on sentence boundaries so each chunk contains complete thoughts. 

import re

def sentence_chunk(text, max_sentences=5, overlap_sentences=1):
    sentences = re.split(r'(?<=[.!?])\s+', text)
    chunks = []
    for i in range(0, len(sentences), max_sentences - overlap_sentences):
        chunk = " ".join(sentences[i:i + max_sentences])
        if chunk.strip():
            chunks.append(chunk)
    return chunks

Better! Each chunk is at least a complete thought. But sentences vary wildly in length — you might get chunks of 20 tokens or 500 tokens.

Strategy 3: Semantic Chunking

The most sophisticated approach: use embeddings to detect where the meaning shifts in the text, and split at those boundaries. 

from openai import OpenAI
import numpy as np

client = OpenAI()

def get_embedding(text):
    response = client.embeddings.create(model="text-embedding-3-small", input=text)
    return response.data[0].embedding

def semantic_chunk(sentences, similarity_threshold=0.8):
    chunks = []
    current_chunk = [sentences[0]]

    for i in range(1, len(sentences)):
        emb_prev = np.array(get_embedding(sentences[i - 1]))
        emb_curr = np.array(get_embedding(sentences[i]))
        similarity = np.dot(emb_prev, emb_curr) / (np.linalg.norm(emb_prev) * np.linalg.norm(emb_curr))

        if similarity >= similarity_threshold:
            current_chunk.append(sentences[i])
        else:
            chunks.append(" ".join(current_chunk))
            current_chunk = [sentences[i]]

    if current_chunk:
        chunks.append(" ".join(current_chunk))
    return chunks

This is expensive — you're making an embedding API call for every sentence. But for high-value corpora (legal contracts, medical records), the quality improvement is worth it.

Comparison Summary

Strategy Speed Quality Cost Best For
Fixed-size Very fast Okay Free Quick prototypes, uniform text
Sentence-aware Fast Good Free General documents, articles
Semantic Slow Excellent Embedding API calls High-value, complex documents

Embedding Model Selection

Not all embedding models are created equal. The model you choose affects retrieval quality, speed, cost, and vector size (which affects storage).

Model Provider Dimensions Cost (per 1M tokens) Notes
text-embedding-3-small OpenAI 1536 $0.02 Great balance of cost and quality. Default choice.
text-embedding-3-large OpenAI 3072 $0.13 Higher quality, 2x the storage.
embed-english-v3.0 Cohere 1024 $0.10 Strong for English. Supports search/classify types.
all-MiniLM-L6-v2 Open Source (HF) 384 Free (local) Fast, tiny, runs locally. Good for prototypes.

Important rule: You must embed your documents and your queries with the same model. If you embed documents with text-embedding-3-small and query with all-MiniLM-L6-v2, the vectors live in completely different spaces and similarity search will return garbage.

Query Rewriting — Making Retrieval Smarter

Here's a problem you'll hit in production: users ask terrible questions. Not terrible as in rude — terrible as in vague, incomplete, or phrased nothing like the documents.

Vikram types: "refund?"

That single word gets embedded and compared against your chunks. But "refund?" as a vector is much less specific than "What is the refund policy for premium members?" — so the retriever returns weakly relevant chunks, and the LLM gives a mediocre answer.

Query rewriting uses the LLM to expand or rephrase the user's question into a better search query before retrieval: 

def rewrite_query(user_question):
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "You are a search query optimizer. Rewrite the user's question into a clear, specific search query that would match relevant document chunks. Output ONLY the rewritten query, nothing else."},
            {"role": "user", "content": user_question}
        ],
        temperature=0.0
    )
    return response.choices[0].message.content

# Example:
original = "refund?"
rewritten = rewrite_query(original)
print(f"Original: {original}")
print(f"Rewritten: {rewritten}")
# Output: "What is the refund policy and how do I request a refund?"

Wait — we're using the LLM just to improve the search query? Yes! This adds one extra API call (cheap with a mini model), but the retrieval quality improvement is dramatic. The rewritten query produces much better embedding vectors, which pull out far more relevant chunks.

The trade-off is latency and cost. In practice, query rewriting adds ~200-500ms and a few cents per 1000 queries. For most production systems, that's a great trade.

# --- Retrieval Quality Measurement ---
def measure_retrieval(collection, eval_set, top_k=3, label=""):
    hits = 0
    for case in eval_set:
        results = collection.query(query_texts=[case["query"]], n_results=top_k)
        retrieved_text = " ".join(results["documents"][0])
        found = any(kw.lower() in retrieved_text.lower() for kw in case["expected_keywords"])
        if found:
            hits += 1
        print(f"  {'✓' if found else '✗'} {case['query'][:50]}...")
    
    score = hits / len(eval_set) * 100
    print(f"\n  [{label}] Retrieval Hit Rate: {hits}/{len(eval_set)} ({score:.0f}%)\n")
    return score
