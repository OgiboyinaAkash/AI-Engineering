Vector Databases, Retrieval Quality & Hybrid Search

In C2.1, we focused on getting documents into a vector database and getting chunks out. That works great for a 50-document knowledge base. But what happens when your company has 10,000 documents across 15 product lines, written over 5 years by different teams?

Suddenly, "find the most similar chunks" isn't good enough. You need to filter by product, by date, by document type. You need to combine keyword search with semantic search. And you absolutely need to measure whether your retrieval is actually working — not just eyeball it.

That's what this module is about.

Vector DB Fundamentals: Local vs Hosted

You've used ChromaDB — a local, in-memory vector database. It's fantastic for development. But in production, you need persistence, scalability, and sometimes multi-tenant access. Let's compare the options:

Feature ChromaDB (Local) Pinecone (Cloud)
Setup pip install chromadb — done. Create account, get API key, configure index.
Persistence Optional (file-based). Data gone if in-memory. Always persistent. Managed cloud storage.
Scale Works well up to ~100K vectors on a single machine. Billions of vectors. Auto-scaling.
Cost Free forever. Free tier available; paid tiers for production.
Best for Development, prototypes, small production apps. Large-scale production with many users.

ChromaDB with Persistence

In C1.5, we used in-memory ChromaDB — data vanished when the script ended. For production, use persistent storage: 

import chromadb

# Persistent storage — data survives restarts
client = chromadb.PersistentClient(path="./chroma_db")

# Get or create a collection (idempotent)
collection = client.get_or_create_collection(
    name="knowledge_base",
    metadata={"hnsw:space": "cosine"}  # Use cosine similarity
)

# Add documents with metadata
collection.add(
    documents=["Premium plan refund window is 60 days.", "Standard plan refund window is 30 days."],
    metadatas=[
        {"source": "refund_policy.pdf", "category": "billing", "year": 2026},
        {"source": "refund_policy.pdf", "category": "billing", "year": 2026}
    ],
    ids=["ref_001", "ref_002"]
)

print(f"Collection has {collection.count()} documents.")

Notice the metadatas parameter? That's new. Each document now carries structured metadata alongside its embedding. This is the key to the next section.

Metadata Filtering — Precision Beyond Semantics

Here's a real production problem: Priya works at a company with 50 product lines. She asks the RAG system: "What's the pricing for the Pro plan?" The retriever returns chunks about "Pro plan" pricing — but from 3 different products. The LLM mashes them together into a confused answer that's partially wrong for all of them.

Metadata filtering solves this. Instead of searching the entire vector space, you first narrow down to a subset using structured filters (product name, date, category) and then do semantic search within that subset. 

# Add documents with rich metadata
collection.add(
    documents=[
        "CloudNest Pro hosting costs ₹1,499/month.",
        "CloudNest Pro email costs ₹299/month.",
        "CloudNest Pro storage costs ₹699/month.",
        "CloudNest Starter hosting costs ₹499/month.",
    ],
    metadatas=[
        {"product": "hosting", "plan": "pro", "type": "pricing"},
        {"product": "email", "plan": "pro", "type": "pricing"},
        {"product": "storage", "plan": "pro", "type": "pricing"},
        {"product": "hosting", "plan": "starter", "type": "pricing"},
    ],
    ids=["h_pro", "e_pro", "s_pro", "h_starter"]
)

# Query WITH metadata filter — only search within hosting products
results = collection.query(
    query_texts=["What does the Pro plan cost?"],
    where={"product": "hosting"},  # Filter BEFORE semantic search
    n_results=2
)

print(results["documents"])
# Only returns hosting-related Pro plan pricing, not email or storage

Think of it like this: Without metadata filtering, you're searching a library with no sections — every book is mixed together. With metadata filtering, you walk to the "Hosting" shelf first, then look for "Pro plan pricing" within that shelf. Much more precise.

ChromaDB supports powerful filter operators: 

# AND: Multiple conditions
results = collection.query(
    query_texts=["pricing"],
    where={"$and": [{"product": "hosting"}, {"plan": "pro"}]},
    n_results=3
)

# OR: Match any condition
results = collection.query(
    query_texts=["pricing"],
    where={"$or": [{"plan": "pro"}, {"plan": "enterprise"}]},
    n_results=3
)

# Comparison: Filter by year
results = collection.query(
    query_texts=["refund policy"],
    where={"year": {"$gte": 2025}},  # Only docs from 2025 onwards
    n_results=3
)

Hybrid Search & Reranking

Semantic search is great at understanding meaning but can miss exact terms. Keyword search is great at exact matches but has zero understanding of meaning. Hybrid search combines both.

Consider this: a user asks for "error code ERR-4521". Semantic search might return chunks about "errors" and "troubleshooting" in general — because those are semantically similar. But the user wants the specific error code. Keyword search would find ERR-4521 exactly. Hybrid search does both and merges the results.

Simple Hybrid Search Implementation

def keyword_search(documents, query, top_k=3):
    """Simple BM25-like keyword matching"""
    query_terms = query.lower().split()
    scores = []
    for i, doc in enumerate(documents):
        doc_lower = doc.lower()
        score = sum(1 for term in query_terms if term in doc_lower)
        scores.append((i, score))
    scores.sort(key=lambda x: x[1], reverse=True)
    return [documents[i] for i, _ in scores[:top_k]]

def hybrid_search(collection, documents, query, top_k=3):
    """Combine semantic (vector) and keyword search results"""
    # Semantic results
    semantic_results = collection.query(query_texts=[query], n_results=top_k)
    semantic_docs = set(semantic_results["documents"][0])
    
    # Keyword results
    keyword_docs = set(keyword_search(documents, query, top_k))
    
    # Merge: prioritize docs found by BOTH methods
    both = semantic_docs & keyword_docs
    semantic_only = semantic_docs - keyword_docs
    keyword_only = keyword_docs - semantic_docs
    
    merged = list(both) + list(semantic_only) + list(keyword_only)
    return merged[:top_k]

Reranking — The Quality Multiplier

After retrieving candidate chunks (from semantic, keyword, or hybrid search), you can run a reranker — a model that scores how well each chunk actually answers the query. The reranker re-orders the chunks by relevance quality, not just embedding similarity.

A simple LLM-based reranker: 

def rerank_with_llm(query, chunks, top_k=3):
    """Use LLM to score relevance of each chunk to the query"""
    scored = []
    for chunk in chunks:
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[
                {"role": "system", "content": "Rate how relevant the following document chunk is to the query. Respond with ONLY a number from 0 to 10."},
                {"role": "user", "content": f"Query: {query}\n\nChunk: {chunk}"}
            ],
            temperature=0.0
        )
        try:
            score = float(response.choices[0].message.content.strip())
        except ValueError:
            score = 0
        scored.append((chunk, score))
    
    scored.sort(key=lambda x: x[1], reverse=True)
    return [chunk for chunk, _ in scored[:top_k]]

This is expensive — one LLM call per chunk per query. In production, you'd use a specialised cross-encoder model (like cross-encoder/ms-marco-MiniLM-L-6-v2) that's much faster and cheaper. But the LLM-based approach works for learning and small-scale systems.

Retrieval Quality Metrics

"My retrieval works fine — I tested it on 3 queries and it looked good."

That's not engineering. To build reliable RAG systems, you need quantitative metrics against a labeled evaluation set. Here are the metrics that matter:

Metric What It Measures Formula (Intuitive)
Recall@k Of all the relevant chunks that exist, how many did we find in the top-k results? relevant_found / total_relevant
Precision@k Of the k results we returned, how many are actually relevant? relevant_found / k
MRR (Mean Reciprocal Rank) How high up in the results is the first relevant chunk? Average of 1/rank_of_first_relevant across queries

Here's how to compute them:

def compute_retrieval_metrics(eval_set, collection, top_k=5):
    """
    eval_set: list of {"query": str, "relevant_ids": [str]}
    """
    recalls = []
    precisions = []
    reciprocal_ranks = []
    
    for case in eval_set:
        results = collection.query(query_texts=[case["query"]], n_results=top_k)
        retrieved_ids = results["ids"][0]
        relevant_ids = set(case["relevant_ids"])
        
        # Recall@k
        found = len(relevant_ids & set(retrieved_ids))
        recall = found / len(relevant_ids) if relevant_ids else 0
        recalls.append(recall)
        
        # Precision@k
        precision = found / top_k
        precisions.append(precision)
        
        # MRR
        rr = 0
        for rank, rid in enumerate(retrieved_ids, 1):
            if rid in relevant_ids:
                rr = 1 / rank
                break
        reciprocal_ranks.append(rr)
    
    print(f"Recall@{top_k}:    {sum(recalls)/len(recalls):.3f}")
    print(f"Precision@{top_k}: {sum(precisions)/len(precisions):.3f}")
    print(f"MRR:            {sum(reciprocal_ranks)/len(reciprocal_ranks):.3f}")

What good numbers look like: For a well-tuned RAG system, Recall@5 above 0.85 and MRR above 0.7 are solid targets. If your Recall@5 is below 0.6, your retrieval has a serious problem — the LLM is flying blind on 40% of queries.
