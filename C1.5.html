RAG & Knowledge Grounding

The Problem — Why LLMs Make Things Up

Picture this: your company just shipped a customer support chatbot powered by an LLM. It's live, handling thousands of queries a day. One morning, a customer types: "What's your refund policy for orders placed after March 2026?"

The chatbot confidently replies: "Orders placed after March 2026 are eligible for a full refund within 45 days, with a 10% restocking fee for opened items."

Sounds professional, right? One problem: your company never had that policy. The 45 days, the 10% fee — all made up. The LLM generated text that sounded like a real policy because it's seen thousands of similar policies during training. Now the customer demands their money back based on what the bot told them.

This is the hallucination problem in production. And it's not a bug — it's a fundamental property of how LLMs work. Remember from C1.1: LLMs are next-token predictors. They generate text that sounds right, not text that is right. They've never seen your company's actual policies.

So does that mean we can never trust LLMs with company-specific information?

No! We absolutely can. We just need to give them the right information at query time. Instead of relying on what the LLM memorized during training, we retrieve the actual documents and hand them to the LLM along with the question.

This is the core idea behind Retrieval-Augmented Generation, or RAG.

Think of it like this: a closed-book exam vs an open-book exam.

  • Closed-book (vanilla LLM): You rely purely on memory. And memory can be wrong, incomplete, or outdated. You might confidently write an answer that sounds plausible but is factually wrong.
  • Open-book (RAG): You look up the answer in your textbook first, then write your response based on what you actually found. Much, much more accurate.

RAG turns every LLM query into an open-book exam.

How RAG Works — The Big Picture

The RAG pipeline has two phases: a one-time preparation phase (where you prepare your knowledge base) and a query phase (where you answer questions). Here's the complete flow:

Preparation Phase (done once)

  1. Chunk: Take your knowledge base — company docs, PDFs, wikis, product manuals, HR policies, whatever — and split it into smaller pieces called chunks. Each chunk is typically a paragraph or a few sentences. Small enough to be focused, large enough to retain meaning.
  2. Embed: Convert each chunk into a numerical representation called an embedding — a list of numbers (a vector) that captures the meaning of the text. Two chunks about similar topics will have similar vectors. This is done using an embedding model (like OpenAI's text-embedding-3-small).
  3. Store: Save these vectors in a vector database (like ChromaDB, Pinecone, or FAISS). Think of it as a searchable library indexed by meaning, not by keywords.

Query Phase (every user question)

  1. Embed the question: When a user asks something, convert their question into an embedding using the same model.
  2. Retrieve: Search the vector database for chunks whose embeddings are most similar to the question's embedding. These are the most relevant pieces of your knowledge base.
  3. Generate: Pass the retrieved chunks as context in the prompt to the LLM, along with the user's question. The LLM answers based on the provided context, not from memory.

Wait, so we're literally copy-pasting relevant documents into the prompt?

Yes, essentially! That's the beauty of it. We're using the LLM's reasoning ability to synthesize an answer, but grounding it in actual source material. The LLM doesn't have to remember anything — we hand it the answer sheet.

Here's the flow in one line:

User asks question → Question gets embedded → Vector DB returns top matching chunks → Chunks + question go into the LLM prompt → LLM answers based on chunks

If you remember the tool calling pattern from C1.4 — where we gave the LLM hands to act in the world — RAG gives the LLM eyes to read your documents.

Embeddings — The Secret Sauce

Let's talk about the magic behind RAG: embeddings.

An embedding converts a piece of text into a point in a high-dimensional space. Texts with similar meanings end up near each other in this space. Texts about completely different topics end up far apart.

For example:

  • "The cat sat on the mat" and "A kitten was resting on the rug" → Very close in embedding space (similar meaning), even though they share almost no words.
  • "The cat sat on the mat" and "The stock market crashed today" → Very far in embedding space (completely different topics).

But how does it know that "cat" and "kitten" are similar if they're different words?

The embedding model was trained on massive text corpora where "cat" and "kitten" appear in nearly identical contexts. It learned that they're semantically interchangeable, just like you did when you learned the language. This is why embedding search is called semantic search — it searches by meaning, not by exact keyword matches.

Here's how to create an embedding in code:

from openai import OpenAI
client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="What is the refund policy for premium members?"
)

embedding_vector = response.data[0].embedding
print(f"Embedding dimensions: {len(embedding_vector)}")
print(f"First 5 values: {embedding_vector[:5]}")

Output:

Embedding dimensions: 1536
First 5 values: [0.0023, -0.0121, 0.0456, -0.0089, 0.0234]

Each embedding is a list of 1,536 numbers. Don't worry about what each number means — just know that similar texts produce similar lists of numbers. That's all your code needs to know to find relevant documents.

Cost note: Embedding models are extremely cheap compared to generation models. With text-embedding-3-small, you can embed around 60,000 pages of text for about $1. So don't worry about embedding costs.

Vector Databases & Similarity Search

Once you have embeddings, you need somewhere to store them and search through them efficiently. That's what a vector database is — a database optimized for storing vectors and finding the most similar ones to a query vector.

Here's a good analogy: Imagine a library where books aren't organized by author or title, but by what they're about. Books about cooking Italian pasta are near books about making pizza dough. Books about machine learning are near books about statistics. When you walk in and say "I need something about making homemade pasta", the librarian instantly points you to the right shelf — not because they searched every page, but because they know where the "Italian cooking" region is in the library's meaning-space.

That's exactly how a vector database works. The "shelves" are regions in embedding space, and "pointing you to the right one" is a similarity search.

Let's use ChromaDB — a simple, local vector database that's perfect for learning. It even generates embeddings for you automatically: 

import chromadb

client = chromadb.Client()
collection = client.create_collection(name="company_docs")

# Add documents — ChromaDB will auto-generate embeddings
collection.add(
    documents=[
        "Premium members get a 60-day refund window on all purchases.",
        "Standard members get a 30-day refund window.",
        "All refunds require the original receipt or order confirmation email.",
        "Our server uptime SLA guarantees 99.9% availability.",
        "Contact support@cloudnest.com for billing and payment issues."
    ],
    ids=["doc1", "doc2", "doc3", "doc4", "doc5"]
)

# Search for relevant documents
results = collection.query(
    query_texts=["What is the refund policy for premium users?"],
    n_results=3
)

print("Top 3 relevant documents:")
for i, doc in enumerate(results["documents"][0]):
    print(f"  {i+1}. {doc}")

Expected output:

Top 3 relevant documents:
  1. Premium members get a 60-day refund window on all purchases.
  2. Standard members get a 30-day refund window.
  3. All refunds require the original receipt or order confirmation email.

Notice something? The query said "premium users" but the document said "premium members" — and it still matched perfectly! And the SLA and billing docs were correctly excluded because they're not about refunds. That's semantic search at work.

Compare this to traditional keyword search:

Feature Keyword Search (Ctrl+F) Semantic Search (Embeddings)
"refund policy" matches "return guidelines" No Yes
Speed Very fast Fast (with index)
Understands meaning No Yes
Requires embedding model No Yes

Putting It All Together — The RAG Pipeline

Now let's wire everything together into a complete, working RAG system. We'll load documents into ChromaDB, retrieve relevant chunks for a user's question, and use them to ground the LLM's response.

The most important piece is the prompt template. Pay very close attention to this: 

def build_rag_prompt(context_chunks, user_question):
    context = "\n---\n".join(context_chunks)
    return f"""Answer the user's question based ONLY on the following context.
If the context doesn't contain enough information to answer, say "I don't have information about that in the available documents."
Do not make up or infer information that isn't explicitly stated in the context.

Context:
{context}

Question: {user_question}
Answer:"""

That instruction — "based ONLY on the following context" — is the single most important line in any RAG system. Without it, the LLM will happily mix retrieved facts with hallucinated ones. It'll take one real fact from your documents and then "helpfully" add three made-up details. With this instruction, the LLM stays grounded.

Here's the complete end-to-end code: 

import os
import chromadb
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
llm_client = OpenAI()

# --- Step 1: Load documents into vector database ---
chroma_client = chromadb.Client()
collection = chroma_client.create_collection(name="cloudnest_policies")

policy_docs = [
    "Premium plan customers are eligible for a full refund within 60 days of purchase.",
    "Standard plan customers can request a refund within 30 days of purchase.",
    "Enterprise customers get a 90-day refund window with prorated refunds available up to 6 months.",
    "All refund requests must include the original order ID and registered email address.",
    "CloudNest guarantees 99.9% server uptime under our Service Level Agreement.",
    "Scheduled maintenance windows are announced 48 hours in advance via email.",
    "Technical support is available 24/7 for Premium and Enterprise plans.",
    "Standard plan customers receive support during business hours: 9 AM to 6 PM IST.",
    "Data backups are performed daily and retained for 30 days on all plans.",
    "CloudNest does not share customer data with third parties under any circumstances."
]

collection.add(
    documents=policy_docs,
    ids=[f"doc{i}" for i in range(len(policy_docs))]
)
print(f"Loaded {len(policy_docs)} documents into vector database.\n")

# --- Step 2: Retrieval function ---
def retrieve(question, n_results=3):
    results = collection.query(
        query_texts=[question],
        n_results=n_results
    )
    return results["documents"][0], results["ids"][0]

# --- Step 3: RAG prompt builder ---
def build_rag_prompt(context_chunks, user_question):
    context = "\n---\n".join(context_chunks)
    return f"""Answer the user's question based ONLY on the following context.
If the context doesn't contain enough information to answer, say "I don't have information about that in the available documents."
Do not make up or infer information that isn't explicitly stated in the context.

Context:
{context}

Question: {user_question}
Answer:"""

# --- Step 4: Complete RAG query ---
def ask(question):
    # Retrieve relevant chunks
    chunks, chunk_ids = retrieve(question)
    print(f"Retrieved {len(chunks)} relevant chunks...")

    # Build grounded prompt
    prompt = build_rag_prompt(chunks, question)

    # Query LLM with context
    response = llm_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "You are a helpful assistant for CloudNest Hosting. Answer questions accurately based on provided context."},
            {"role": "user", "content": prompt}
        ],
        temperature=0.0
    )

    answer = response.choices[0].message.content
    print(f"A: {answer}")
    print(f"[Source chunks: {', '.join(chunk_ids)}]\n")
    return answer

# --- Step 5: Test it! ---
if __name__ == "__main__":
    print("--- RAG Q&A Bot ---\n")

    test_questions = [
        "What is the refund policy for enterprise customers?",
        "When is technical support available for standard plan users?",
        "How long are data backups retained?",
        "Does CloudNest offer GPU instances?",
        "What programming languages does CloudNest support?",
    ]

    answerable = 0
    declined = 0

    for q in test_questions:
        print(f"Q: {q}")
        answer = ask(q)
        if "don't have information" in answer.lower():
            declined += 1
        else:
            answerable += 1

    print(f"--- Results ---")
    print(f"Questions answered from docs: {answerable}")
    print(f"Correctly declined: {declined}")

Expected output:

--- RAG Q&A Bot ---

Loaded 10 documents into vector database.

Q: What is the refund policy for enterprise customers?
Retrieved 3 relevant chunks...
A: Enterprise customers get a 90-day refund window with prorated refunds available up to 6 months. All refund requests must include the original order ID and registered email address.
[Source chunks: doc2, doc3, doc0]

Q: When is technical support available for standard plan users?

[Source chunks: doc7, doc6, doc1]

Q: How long are data backups retained?
Retrieved 3 relevant chunks...
A: Data backups are performed daily and retained for 30 days on all plans.
[Source chunks: doc8, doc4, doc5]

Retrieved 3 relevant chunks...
A: I don't have information about that in the available documents.
[Source chunks: doc4, doc6, doc8]

[Source chunks: doc6, doc7, doc4]

Look at that! The bot correctly answered 3 questions using real policy data, and correctly said "I don't know" for 2 questions that aren't covered in the documents. No hallucinations. No made-up policies. That's the power of RAG.

Chunking Strategies

We mentioned earlier that you split your documents into "chunks" before embedding them. This step sounds simple, but how you chunk your documents matters enormously for answer quality.

Think about it: if your chunk is just the fragment "60 days", that tells you nothing useful. 60 days of what? For whom? But if the chunk is "Premium plan customers are eligible for a full refund within 60 days of purchase", now the LLM has everything it needs.

On the flip side, if your chunk is 3 pages long, you're wasting precious context window space and diluting the relevance — the LLM has to find the needle in a mini-haystack within the chunk itself.

Here are the common strategies:

  1. Fixed-size chunking: Split every N characters or N tokens. Simple to implement, works reasonably well. The risk is cutting a sentence in half mid-thought. 

    # Simple fixed-size chunking
def chunk_text(text, chunk_size=500, overlap=50):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start = end - overlap  # Overlap with previous chunk
    return chunks
  2. Sentence-based chunking: Split on sentence boundaries. Preserves complete thoughts, which leads to better embeddings and better retrieval. Slightly more complex to implement.
  3. Overlap chunking: Each chunk overlaps with the previous one by some number of tokens (say 50). This ensures that information sitting at the boundary between two chunks isn't lost. If a key fact spans the boundary, both chunks will contain it.

Rule of thumb: Start with chunks of 200-500 tokens with a 50-token overlap. Adjust based on your results. If the LLM's answers seem to lack context, increase chunk size. If retrieval seems imprecise, decrease it.

You'll hear about libraries like LangChain and LlamaIndex that handle chunking, embedding, and retrieval for you. They're great tools and we'll use them later, but it's important to understand what they're doing under the hood first — which is exactly what we've covered in this chapter.

Lab: Build a RAG-Powered Q&A Bot

1. Problem Setup

You are building an internal Q&A assistant for CloudNest Hosting (continuing from C1.3's scenario). The company has a set of policy documents about refunds, SLAs, support hours, data handling, and pricing. Employees need to quickly find accurate answers without reading through everything.

The critical requirement: the bot must answer ONLY based on the provided documents and clearly say "I don't know" when the information isn't available. No hallucinations allowed.

2. Environment Setup

pip install openai chromadb python-dotenv

3. Step-by-Step Instructions

  1. Define a list of 10 company policy statements as mock data (or use the ones from the section above).
  2. Initialize ChromaDB and load the policy documents.
  3. Write the retrieval function that takes a question and returns the top 3 relevant chunks.
  4. Build the RAG prompt template with the grounding instruction.
  5. Wire everything together: question → retrieve → build prompt → LLM → answer.
  6. Test with questions that ARE in the docs and questions that AREN'T.

4. Starter Code

import os
import chromadb
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
llm_client = OpenAI()

# --- Company Policy Data ---
policies = [
    "CloudNest Starter plan costs ₹499/month and includes 1 vCPU, 1 GB RAM, and 20 GB SSD storage.",
    "CloudNest Pro plan costs ₹1,499/month and includes 2 vCPUs, 4 GB RAM, and 80 GB SSD storage.",
    "CloudNest Enterprise plan costs ₹4,999/month with dedicated resources and a personal account manager.",
    "All plans include free SSL certificates and DDoS protection.",
    "Premium plan customers are eligible for a full refund within 60 days of purchase.",
    "Standard plan customers can request a refund within 30 days of purchase.",
    "Enterprise customers get a 90-day refund window with prorated refunds up to 6 months.",
    "Technical support is available 24/7 for Pro and Enterprise plans via chat and email.",
    "Starter plan support is available during business hours: 9 AM to 6 PM IST, weekdays only.",
    "Data backups are automatic, performed daily, and retained for 30 days across all plans.",
    "CloudNest data centers are located in Mumbai, Bangalore, and Singapore.",
    "Customers can request data deletion at any time by contacting privacy@cloudnest.com."
]

# --- Initialize Vector Database ---
chroma = chromadb.Client()
collection = chroma.create_collection(name="cloudnest_kb")
collection.add(
    documents=policies,
    ids=[f"policy_{i}" for i in range(len(policies))]
)
print(f"Loaded {len(policies)} policy documents.\n")

# --- Retrieval ---
def retrieve_context(question, top_k=3):
    results = collection.query(query_texts=[question], n_results=top_k)
    return results["documents"][0], results["ids"][0]

# --- RAG Prompt ---
def build_prompt(context_docs, question):
    context = "\n---\n".join(context_docs)
    return f"""Answer the question based ONLY on the context below.
If the answer is not in the context, respond with: "I don't have information about that."

Context:
{context}

Question: {question}
Answer:"""

# --- Ask Function ---
def ask(question):
    docs, ids = retrieve_context(question)
    prompt = build_prompt(docs, question)

    response = llm_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "You are CloudNest's internal knowledge assistant. Be concise and accurate."},
            {"role": "user", "content": prompt}
        ],
        temperature=0.0
    )

    answer = response.choices[0].message.content
    return answer, ids

# --- Test Harness ---
if __name__ == "__main__":
    test_cases = [
        {"q": "How much does the Pro plan cost?", "has_answer": True},
        {"q": "What is the refund window for enterprise customers?", "has_answer": True},
        {"q": "Where are CloudNest data centers located?", "has_answer": True},
        {"q": "Does CloudNest support Kubernetes deployments?", "has_answer": False},
        {"q": "Can I get a custom domain with my hosting plan?", "has_answer": False},
    ]

    correct = 0

    for case in test_cases:
        print(f"Q: {case['q']}")
        answer, source_ids = ask(case['q'])
        print(f"A: {answer}")
        print(f"[Sources: {', '.join(source_ids)}]\n")

        is_decline = "don't have information" in answer.lower()
        if case["has_answer"] and not is_decline:
            correct += 1
        elif not case["has_answer"] and is_decline:
            correct += 1

    print(f"--- Accuracy: {correct}/{len(test_cases)} ---")

6. Failure Scenario

Remove the grounding instruction from the prompt. Replace the build_prompt function's instructions with just: "Answer this question: {question}" (without context or the ONLY instruction). Now ask: "Does CloudNest support Kubernetes deployments?"

Watch how the LLM now hallucates a plausible-sounding answer: "Yes, CloudNest offers managed Kubernetes clusters starting at..." — completely fabricated. This demonstrates exactly why the grounding instruction is non-negotiable in production RAG systems.
